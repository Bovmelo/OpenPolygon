# Stressor Stress

## Purpose

This skill performs differential testing between an optimized solution and a brute force / reference solution for ICPC/CCPC style problems.

It compares optimized output against brute output through a testlib checker generated by `stressor-checker`.

This skill does not use diff.

This skill does not use `diff -w`.

This skill does not use exact textual comparison as fallback.

The checker is invoked as:

```bash
./build/checker input_file optimized.out brute.out
```

Here:

* `input_file` is the original input.
* `optimized.out` is treated as participant output / `ouf`.
* `brute.out` is treated as jury answer / `ans`.

The testing order is strict:

1. `tests/small`
2. `tests/strong`
3. generated `tests/large`

If early-stage tests fail, stop immediately and report.

Do not continue to larger generated tests when smaller trusted tests already expose bugs.

## Scope

Allowed output when this skill is invoked:

```text
scripts/stress.sh
stress report markdown file placed next to the optimized solution file
outputs/stress/
tests/strong/                 updated only by copying failing large cases into it
tests/large/                  temporary generated large data, cleaned at the end
build/                        created by scripts if needed
```

The stress report is **not** written to a dedicated `reports/` directory. It
lives in the same directory as the optimized solution under test (see
"Failure Report Requirement" below for the exact filename rule). The
preferred form for a versioned optimized solution is
`solutions/stress_report_<optimized_version>_<report_id>.md`.

Forbidden output unless explicitly asked:

```text
solutions/brute.cpp
solutions/optimize_XXXX.cpp
validators/val.cpp
generators/*.cpp
checkers/checker.cpp
checker.cpp
answers/
new optimized solutions
new brute solutions
new validators
new generators
new checkers
```

This skill must not write optimized code.

This skill must not write brute code.

This skill must not write validators.

This skill must not write generators.

This skill must not write checkers.

This skill only orchestrates existing files.

Preferred invocation:

```text
/stressor-stress path_to_problem_statement
```

Optional arguments:

```text
/stressor-stress path_to_problem_statement --optimized latest

/stressor-stress path_to_problem_statement --brute solutions/brute.cpp

```

Defaults:

* `--optimized latest`: use the latest `solutions/optimize_XXXX.cpp`
* `--brute solutions/brute.cpp`
* `--rounds 10`: number of large generation rounds or generated groups if generator supports it

If `--optimized latest` is used:

* scan `solutions/optimize_XXXX.cpp`
* choose the largest XXXX
* do not use `solutions/optimized.cpp` unless no versioned optimized file exists and the user explicitly allows it

If no optimized solution is found, stop and report.

If no brute solution is found, stop and report.

If no checker is found, stop and report.

## Official Sample Discovery

After resolving the statement directory, search for official sample files under the statement directory.

The preferred statement directory is:

```text
{problemname}/statement/
```

Official sample file pairs may include:

```text
*.in  + matching *.out
*.input + matching *.output
sample*.in + sample*.out
example*.in + example*.out
ex*.in + ex*.out
```

Examples:

```text
QOJ464/statement/ex_prefix1.in
QOJ464/statement/ex_prefix1.out
```

Pairing rule:

* A sample input file `X.in` pairs with `X.out`.
* A sample input file `X.input` pairs with `X.output`.
* Preserve lexical order when listing sample pairs.
* Do not guess outputs if no matching output file exists.
* If an input sample has no output sample, it may still be used for validator checks, but not for brute-vs-official-output checks.
* If an output sample has no matching input sample, report it as orphaned and do not use it.

Sample files are official statement content.

They may be read by skills that are allowed to read the statement bundle.

They are not generated tests.

They should not be modified.

Forbidden:

* Do not copy statement sample files into tests/strong automatically.
* Do not overwrite statement sample files.
* Do not treat generated tests as official samples.
* Do not infer hidden rules only from sample outputs.

## Official Samples and Stress

Statement samples under `statement/` are **not** automatically part of the stress test set unless they are explicitly copied or exported into a tests directory.

The default stress stage order (`tests/small` → `tests/strong` → generated `tests/large`) is unchanged. `stressor-init` may run an official sample preflight separately; that is not this skill's job.

If the user explicitly asks `stressor-stress` to include official samples, then for each pair `(statement/X.in, statement/X.out)`:

* use validator if available (and skip the sample if it falls outside `VALIDATION_SCOPE.md`);
* run brute on `X.in` if `X.in` is within `solutions/BRUTE_LIMITS.json`; otherwise skip and report `SAMPLE_OUT_OF_BRUTE_SCOPE`;
* compare via checker. The preferred invocation when an official sample output exists:

  ```bash
  ./build/checker sample.in optimized.out sample.out
  ```

  Otherwise, compare optimized against brute using checker:

  ```bash
  ./build/checker sample.in optimized.out brute.out
  ```

Do not change the default stress stage order. Official samples are an additive, opt-in stage; never silently insert them into `tests/small/` or `tests/strong/`.

## Checker-Based Comparison Rules

This skill must compare outputs using the testlib checker generated by `stressor-checker`.

Do not use `diff`.

Do not use `diff -w`.

Do not use exact textual comparison.

Do not fallback to diff if checker is missing.

The checker must be compiled from one of:

1. `checkers/checker.cpp`
2. `checker.cpp`

Preferred path:

```text
checkers/checker.cpp
```

Resolve the source path into the shell variable `CHECKER_SRC`, then compile with the testlib include directory discovered below (`TESTLIB_INCLUDE`):

```bash
g++ -std=c++17 -O2 -pipe -I"$TESTLIB_INCLUDE" "$CHECKER_SRC" -o build/checker
```

Do **not** hard-code `-I.` — the project may keep `testlib.h` outside the current directory, including one or two directories above (multi-problem repos).

The checker is invoked as:

```bash
./build/checker input_file optimized.out brute.out
```

The argument order follows standard testlib checker convention:

1. input file
2. participant output
3. jury answer

In this stress workflow:

* participant output = optimized output
* jury answer = brute output

A test is considered passed only if checker exits successfully.

A test is considered failed if checker exits with nonzero status.

For every failed case, save:

* input file
* brute output
* optimized output
* checker log
* checker exit status

The checker result is the source of truth.

If `checker.cpp` is missing, stop and report that `stressor-checker` must be run first.

If `checker.cpp` fails to compile, stop and report the compile error.

If checker itself appears inconsistent, crashes, or times out, report it as `CHECKER_ERROR` or `CHECKER_TIMEOUT`, not as an optimized wrong answer.

## testlib.h Search

Because `checker.cpp` is expected to use testlib, search for `testlib.h` before compiling the checker. Take the first match from these project-local paths:

```text
./testlib.h
./checkers/testlib.h
./third_party/testlib/testlib.h
./third_party/testlib.h
./lib/testlib.h
./generators/testlib.h
../testlib.h
../third_party/testlib.h
../third_party/testlib/testlib.h
../../third_party/testlib.h
../../third_party/testlib/testlib.h
```

The last entries cover multi-problem layouts where `testlib.h` is shared one or two directories above the current problem.

When `testlib.h` is found, set:

```bash
TESTLIB_INCLUDE="<directory-containing-testlib.h>"
```

Examples:

```bash
# if ./third_party/testlib/testlib.h exists
TESTLIB_INCLUDE="./third_party/testlib"

# if ../third_party/testlib/testlib.h exists
TESTLIB_INCLUDE="../third_party/testlib"
```

Compile the checker using both `TESTLIB_INCLUDE` and the resolved `CHECKER_SRC`:

```bash
g++ -std=c++17 -O2 -pipe -I"$TESTLIB_INCLUDE" "$CHECKER_SRC" -o build/checker
```

If `testlib.h` is missing and the checker cannot compile:

* stop
* write report
* do not run tests
* do not download testlib
* do not fallback to diff

## Brute Limits

If `solutions/BRUTE_LIMITS.json` exists:

* read it
* respect its recommended limits
* do not generate or run large tests that exceed brute-supported range
* use its notes to choose safe generated tests

If the user asks for generated tests that the brute cannot run, clamp the size or stop and report.

If `BRUTE_LIMITS.json` is missing:

* use conservative assumptions
* warn that brute runnable range is unknown
* avoid aggressive large tests
* prefer only small and strong tests

Never run brute on data that is obviously outside its documented supported range.

## Compatible Range From config.yaml

If `lightcp/problems/<pid>/config.yaml` exists, read its `x-stressor.compatible_range` block. This block is the canonical safe range for generator-based differential testing because `stressor-init` computed it as the conservative intersection of validator scope, brute limits, checker capability, generator capability, and the statement.

Rules:

* When `compatible_range.status` is `READY`, use those limits as the **final** upper bound for any data sent to large-stage generators. Do not ask generators for tests outside `compatible_range` when brute comparison is required.
* When `compatible_range.status` is `INCOMPLETE` or `CONFLICT`, fall back to `solutions/BRUTE_LIMITS.json` ∩ `validators/VALIDATION_SCOPE.md`, and record in the stress report that `config.yaml` range was unavailable.
* When `config.yaml` is missing entirely, fall back to `BRUTE_LIMITS.json` and `VALIDATION_SCOPE.md` directly. Report that no config-derived range was used.
* When all range sources are missing, skip large generation and only run `tests/small` and `tests/strong`. Report that large stress is unavailable.

The compatible-range upper bound applies only to generator-based stress in Stage 3. It does not constrain `tests/small` or `tests/strong` — those are run as-is.

## Strict Testing Order

The testing order is mandatory.

### Stage 1: tests/small

First test all files in:

```text
tests/small/
```

If `tests/small/` does not exist or contains no files:

* report that small tests are missing
* continue to `tests/strong/` only if the user explicitly wants best-effort behavior
* otherwise stop and request small tests

If any `tests/small` case fails:

* save optimized output and brute output
* save checker log
* create failure details in the report
* do not test `tests/strong`
* do not generate `tests/large`
* do not call generators
* end immediately

### Stage 2: tests/strong

Only run this stage if all `tests/small` cases pass.

Test all files in:

```text
tests/strong/
```

If `tests/strong/` does not exist, create it as an empty directory only if needed for future failure promotion.

If it contains no files, report that no strong tests were available and continue to large generation if generator exists.

If any `tests/strong` case fails:

* save optimized output and brute output
* save checker log
* record exactly which strong tests failed
* do not generate `tests/large`
* end immediately

The final report must clearly list:

```text
Current optimized solution fails on these tests/strong cases:
- tests/strong/...
```

### Stage 3: generated tests/large

Only run this stage if:

* all `tests/small` passed
* all existing `tests/strong` passed

Then generate tests into:

```text
tests/large/
```

Generation must proceed from smaller scale to larger scale.

Use existing generators from `stressor-gen-large`.

Preferred generation entry point:

```bash
bash generators/gen_all_large.sh N tests/large
```

If `generators/gen_all_large.sh` does not exist:

* report that large generation is unavailable
* do not invent a generator
* finish report using only small and strong results

If large generator exists:

* clean or prepare `tests/large/` before generation to avoid mixing stale data
* generate a small batch first
* run optimized vs brute using checker on generated tests
* then increase scale gradually if supported
* stop when failure is found or requested rounds are complete

The generator should be called with sizes that respect `BRUTE_LIMITS.json`.

Do not generate data too large for the brute.

This skill is for differential testing against brute, not for validating optimized solution on full official limits.

### Failing-case promotion

When a generated `tests/large` case fails checker-based comparison:

* copy the failing input file into `tests/strong/`
* give it a fresh, non-colliding filename (e.g. derive a numeric suffix that is one larger than the current maximum in `tests/strong/`, or append the failing batch id)
* never overwrite an existing `tests/strong` file
* record the promotion in the report (which original `tests/large` filename mapped to which new `tests/strong` filename)

Promotion happens within Stage 3, *before* cleanup. If multiple cases in a single batch fail, promote all of them.

### Cleanup after large stage

After all large-stage activity in this run (whether the stage reached the requested number of rounds, hit a failure, or ran out of generator capacity), clean the temporary large data:

```text
tests/large/
```

The cleanup should remove the temporary generated large test files from this run.

Do not delete:

```text
tests/small/
tests/strong/
solutions/
generators/
checkers/
reports/
outputs/
```

Prefer:

```bash
find tests/large -type f -delete
```

Do not delete the `tests/large/` directory itself unless the user explicitly asks.

If the run stops before large generation, do not delete unrelated pre-existing large files unless this skill created or prepared them in this run.

If this skill cleaned `tests/large/` before generation, record that action in the report.

## Script Requirement

Create:

```text
scripts/stress.sh
```

The script should implement the core orchestration.

The agent may also run commands manually if needed, but `scripts/stress.sh` should be reusable.

The script must:

* locate brute
* locate optimized
* locate checker
* locate testlib include path if needed
* compile brute
* compile optimized
* compile checker
* run small tests first
* stop on small failure
* run strong tests second
* stop on strong failure
* generate large tests only after small and strong pass
* run generated large tests
* compare every case using checker
* copy failing large tests to strong
* write outputs and checker logs
* clean tests/large at the end of large-stage run
* never use diff

Suggested interface:

```bash
bash scripts/stress.sh
bash scripts/stress.sh --optimized latest --brute solutions/brute.cpp --rounds 10
bash scripts/stress.sh --optimized solutions/optimize_0003.cpp --brute solutions/brute.cpp --rounds 20
```

The script should be customized to the actual project layout.

Do not leave important parts as TODO placeholders.

The script should use a case runner like:

```bash
run_case() {
    local input="$1"
    local stage="$2"
    local name
    name=$(basename "$input")

    local case_dir="$OUT_DIR/$stage/$name"
    mkdir -p "$case_dir"

    local brute_out="$case_dir/brute.out"
    local opt_out="$case_dir/optimized.out"
    local checker_log="$case_dir/checker.log"

    echo "Running brute on $input"
    timeout "$TIMEOUT_LIMIT" ./build/brute < "$input" > "$brute_out"

    echo "Running optimized on $input"
    timeout "$TIMEOUT_LIMIT" ./build/optimized < "$input" > "$opt_out"

    echo "Checking $input"
    if timeout "$TIMEOUT_LIMIT" ./build/checker "$input" "$opt_out" "$brute_out" > "$checker_log" 2>&1; then
        echo "PASS $input"
        return 0
    else
        echo "FAIL $input"
        return 1
    fi
}
```

The final script should also distinguish:

* optimized wrong answer
* brute runtime error
* optimized runtime error
* checker compile error
* checker runtime error
* checker timeout
* unsupported brute input

## Failure Report Requirement

`reports/` directory.

* If the optimized file is `solutions/optimize_XXXX.cpp`, write the report at
  `solutions/stress_report_XXXX_REPORTID.md`, where `XXXX` is the optimized
  version number parsed from the filename and `REPORTID` is a fresh
  four-digit number or timestamp.
* If the optimized file does not match the `optimize_XXXX.cpp` pattern (for
  example a user-supplied path like `custom/my_solution.cpp`), write the
  report at `<optimized_dir>/stress_report_REPORTID.md`. The report's body

## LightCPVerifier Compatibility

This workflow supports exporting judge-ready assets to a LightCPVerifier-compatible problem directory.

The target directory is:

```text
lightcp/problems/<pid>/
```

Expected structure:

```text
lightcp/problems/<pid>/
├── config.yaml
├── statement.txt
├── checker.cpp
└── testdata/
    ├── small_0001.in
    ├── small_0001.ans
    ├── small_0002.in
    ├── small_0002.ans
    ├── strong_0001.in
    ├── strong_0001.ans
    └── ...
```

Rules:

* `statement.txt` is the plain-text problem statement used by LightCPVerifier.
* `checker.cpp` is copied from `checkers/checker.cpp`.
* `testdata/*.in` are input files.
* `testdata/*.ans` are jury answer files generated by `solutions/brute.cpp`.
* Never generate `.ans` files from optimized attempts.
* Never use `solutions/optimize_XXXX.cpp` as jury answer generator.
* The exported `config.yaml` must explicitly list cases when possible, instead of relying only on `n_cases`.
* The exported package should be suitable for LightCPVerifier `/problem/setup` and `/submit` workflows.

Preferred `config.yaml` shape:

```yaml
type: default
time_limit: 1s
memory_limit: 256m
checker: checker.cpp
filename: main.cpp
subtasks:
  - score: 100
    cases:
      - input: small_0001.in
        output: small_0001.ans
      - input: small_0002.in
        output: small_0002.ans
```

If the problem has known time or memory limits in the statement, use them.

If not known, use conservative defaults and report assumptions:

```yaml
time_limit: 2s
memory_limit: 256m
```

The LightCPVerifier export is a mirror for judge evaluation.

The source-of-truth development directories remain:

```text
tests/small/
tests/strong/
tests/large/
solutions/
checkers/
validators/
generators/
```

### Role of `stressor-stress` in the export

* `stressor-stress` keeps using `tests/small/`, `tests/strong/`, and temporary `tests/large/` for differential testing. Those directories remain the source of truth.
* `stressor-stress` may **promote** a failing generated large case into `tests/strong/` (with a fresh non-colliding filename). That promoted file is now a regression input; it is **not** automatically copied into `lightcp/problems/<pid>/testdata/`. Generating the matching `.ans` and adding the case to the LightCPVerifier package is the job of the export stage (owned by `stressor-init`).
* When the LightCPVerifier export is rerun after a promotion, `tests/strong/<new>` is read and mapped as e.g. `lightcp/problems/<pid>/testdata/strong_00NN.in`; the matching `.ans` is then produced by `solutions/brute.cpp` and the case is appended to `config.yaml`'s explicit case list.
* `stressor-stress` must not regenerate or overwrite the LightCPVerifier package mid-run.

Strict rules for this skill in the LightCPVerifier context:

* `stressor-stress` must **not** write `config.yaml`.
* `stressor-stress` must **not** write `.ans` files into `lightcp/problems/<pid>/testdata/`.
* `stressor-stress` must **not** copy `checker.cpp` or `statement.txt` into a LightCPVerifier package.
* `stressor-stress` must **not** create or modify `lightcp/problems/<pid>/` directly. Promotions are recorded in `tests/strong/` and surfaced in the stress report; the user (or a follow-up `stressor-init` export run) handles the LightCP refresh.

### Official Sample Files in LightCPVerifier Export

Official sample files under `statement/` are export candidates, but `.ans` files for LightCPVerifier should normally be generated by `solutions/brute.cpp`.

Do not blindly use `statement/*.out` as exported `.ans`.

Allowed uses of official sample `.out`:

* validating brute output through checker
* sanity checking checker behavior
* documenting sample consistency

When exporting official samples into `lightcp/problems/<pid>/testdata`, prefer:

```text
statement/ex_prefix1.in -> testdata/sample_0001.in
brute output             -> testdata/sample_0001.ans
```

Then optionally verify:

```bash
checker sample_0001.in sample_0001.ans original_ex_prefix1.out
```

if the checker supports it.

`stressor-stress` does not export samples and does not write into the LightCP package. If the user explicitly asks this skill to include official samples in a stress run, that is purely an additive stress stage as documented in "Official Samples and Stress" above.

## Required Workflow

### 1. Inspect project

Read:

* problem statement path
* `solutions/BRUTE_LIMITS.json`
* `solutions/brute.cpp`
* latest or specified optimized solution
* checker.cpp generated by `stressor-checker`
* `testlib.h` for compiling checker
* existing `tests/small`
* existing `tests/strong`
* generators under `generators/`

Do not modify code yet.

### 2. Locate and compile checker

Search for checker.cpp in this order:

1. `checkers/checker.cpp`
2. `checker.cpp`

Bind the resolved path to `CHECKER_SRC`.

If no checker is found:

* stop
* write report
* do not run tests
* do not fallback to diff

Search for `testlib.h` in the project (see the "testlib.h Search" section above) and bind the discovered include directory to `TESTLIB_INCLUDE`. Compile with:

```bash
g++ -std=c++17 -O2 -pipe -I"$TESTLIB_INCLUDE" "$CHECKER_SRC" -o build/checker
```

Compile checker with the correct include path.

If checker compilation fails:

* stop
* write report
* do not run tests

### 3. Use checker-based comparison

Always use checker-based comparison.

Do not use diff.

Run each case as:

1. Run brute:

```bash
./build/brute < input > brute.out
```

2. Run optimized:

```bash
./build/optimized < input > optimized.out
```

3. Run checker:

```bash
./build/checker input optimized.out brute.out
```

If checker returns success, the case passes.

If checker returns nonzero, the case fails.

### 4. Create or update scripts/stress.sh

Write reusable script for stress orchestration.

### 5. Run Stage 1: tests/small

Compile and run.

If any failure:

* write report
* stop

### 6. Run Stage 2: tests/strong

Only if Stage 1 passes.

If any failure:

* write report
* stop

### 7. Run Stage 3: tests/large

Only if Stage 1 and Stage 2 pass.

Use generator progressively.

Respect brute limits.

Promote failing large cases to tests/strong.

### 8. Cleanup

Clean temporary tests/large files from this run.

### 9. Final report

Generate report and summarize.

## Handling Missing Directories

If `tests/small/` is missing:

* report missing small tests
* do not silently skip unless user explicitly asked for best-effort

If `tests/strong/` is missing:

* create it if needed
* report that no existing strong tests were available

If `tests/large/` is missing:

* create it for temporary generated data

If `generators/gen_all_large.sh` is missing:

* skip Stage 3 with clear report
* do not create generators

## Handling Optimized Versions

If optimized is `latest`:

* scan `solutions/optimize_XXXX.cpp`
* choose the largest version number
* report chosen file

If the user specifies an optimized file:

* use that file
* ensure it exists

Do not overwrite optimized files.

Do not create new optimized files.

## Handling Brute

Default brute:

```text
solutions/brute.cpp
```

If missing:

* stop and report
* do not fabricate brute

If brute refuses an input as unsupported:

* treat that test as unsupported, not as optimized failure
* record it in report
* avoid generating larger data of that type

If many generated tests are unsupported by brute:

* reduce generation size if possible
* otherwise stop large stage and report

## Handling Large Generation Scale

Large generation should proceed from smaller to larger data.

Examples:

```text
round 1: very small / pilot-like data within brute limits
round 2: small-medium data
round 3: medium data
round 4: larger but still within brute limits
```

Do not jump directly to maximum constraints if brute cannot handle them.

Use `BRUTE_LIMITS.json` as the upper bound for generated stress data.

This skill is for differential testing, not for validating optimized solution on full official limits.

## Final Response Format

After using this skill, final response must contain:

```text
Stress summary:
- optimized: ...
- brute: ...
- checker: ...
- comparison method: testlib checker
- checker invocation: checker input optimized.out brute.out
- diff used: no

Stage results:
- tests/small: PASS / FAIL / MISSING
- tests/strong: PASS / FAIL / EMPTY
- tests/large: PASS / FAIL / SKIPPED / UNSUPPORTED

Current failures in tests/strong:
- ...

Promoted failures:
- ...

Checker logs:
- ...

Report:
- path next to optimized solution, for example solutions/stress_report_0003_0001.md

Cleanup:
- tests/large cleaned: yes/no

Next recommended action:
- ...
```

If small tests fail, say clearly:

```text
Stopped after tests/small because small tests already found failures.
```

If strong tests fail, say clearly:

```text
Stopped after tests/strong because existing strong tests found failures.
```

If large tests found failures, say clearly:

```text
Failing generated large tests were copied into tests/strong for future priority testing.
```

If checker is missing, say clearly:

```text
checker.cpp not found. Run stressor-checker first.
```

## Strict Rules

* Must use checker.cpp generated by stressor-checker for comparison.
* Must not use diff.
* Must not use diff -w.
* Must not use exact textual comparison as fallback.
* Must not fallback to diff if checker is missing.
* Must stop and report if checker.cpp is missing.
* Must compile checker before testing.
* Must invoke checker as: checker input optimized_output brute_output.
* Must treat optimized output as participant output.
* Must treat brute output as jury answer.
* Must save checker logs for failed cases.
* Must report CHECKER_ERROR separately from optimized wrong answer if checker crashes or fails unexpectedly.
* Must test `tests/small` first.
* If `tests/small` fails, stop immediately.
* Must test `tests/strong` second.
* If `tests/strong` fails, stop before large generation.
* Must generate `tests/large` only after small and strong pass.
* Must respect `solutions/BRUTE_LIMITS.json` if it exists.
* Must not run brute on data clearly outside its supported range.
* Must not write optimized code.
* Must not write brute code.
* Must not write validator.
* Must not write generator.
* Must not write checker.
* Must not create answer files as official answers.
* Must copy failing large tests into `tests/strong`.
* Must not overwrite existing `tests/strong` files.
* Must generate a stress report.
* Must list current failures in `tests/strong`.
* Must clean `tests/large` after large-stage testing.
* Must not delete `tests/small`, `tests/strong`, `solutions`, `generators`, `checkers`, `reports`, or `outputs`.
* Must not silently ignore failed tests.
* Must not claim PASS if checker was missing or comparison was not run.
* Stress report must be written next to the optimized solution file.
* Do not write stress reports to a separate `reports/` directory by default.
* If optimized file is `solutions/optimize_XXXX.cpp`, name the report as `solutions/stress_report_XXXX_REPORTID.md`.
* Do not overwrite existing stress reports.
* If optimized path is custom, write the report in the optimized file's directory and note in the body that no version number was parsed.
* When `lightcp/problems/<pid>/config.yaml` exists and `x-stressor.compatible_range.status` is `READY`, generator-based Stage 3 tests must respect that range; fall back to `BRUTE_LIMITS.json` ∩ `VALIDATION_SCOPE.md` only when the config range is unavailable, and record which source was used in the stress report.
