# Stressor Polygon Release

## Purpose

This is the top-level problem-setting skill.

Use it when the user gives a problem idea or partial specification and wants the whole contest problem built, verified, and uploaded to Codeforces Polygon.

This skill coordinates:

- `stressor-gen-small`
- `stressor-vali`
- `stressor-brute`
- `stressor-checker`
- `stressor-gen-large`
- `stressor-stress`
- `polygon-mcp`

It is not a replacement for those skills. It is the gatekeeper that decides when each component is ready and when Polygon state is safe to modify.

## Required Discipline

Do not claim the problem is finished until both local verification and Polygon verification have actually been run, or explicitly state what remains unverified.

Do not hide blockers behind compatibility fallbacks.

Do not make large sets of official tests by manually saving concrete inputs. Manual tests are for statement samples and a small number of explainable smoke cases only. Official non-sample tests must be generated by Polygon scripts and generators unless the user explicitly requests otherwise.

Do not use Python for problem code, validators, checkers, generators, or required verification scripts unless the user explicitly asks for Python. Prefer C++ and shell commands around compiled binaries.

## Invocation

Preferred invocation:

```text
$stressor-polygon-release problem_idea_or_statement --polygon-id 123456
```

Common forms:

```text
$stressor-polygon-release "user problem idea" --polygon-id 123456
$stressor-polygon-release ./statement/statement.md --polygon-id 123456
$stressor-polygon-release . --polygon-id 123456 --check-only
$stressor-polygon-release . --polygon-id 123456 --repair polygon
```

If no Polygon id or URL is available, complete the local workflow and stop before remote writes.

## Output Target

A completed project should contain, as applicable:

```text
statement/
solutions/
validators/
checkers/
generators/
tests/small/
tests/strong/
reports/
```

For Polygon release readiness, the remote problem should contain:

```text
statement in the required language slots
resource/statements.ftl when Chinese statement rendering is needed
validator source and selected validator
checker source and selected checker
main accepted solution tagged MA
wrong/TL/RE solutions when useful
small manual samples only
generated official tests created by script lines
validator tests
checker tests
general tutorial
ready package after verified build
```

## Phase 0: Understand And Freeze The Spec

Before writing code, turn the user's idea into a concrete contest specification:

- problem statement
- input format
- output format
- constraints
- legality of edge cases
- construction or optimization target
- uniqueness or non-uniqueness of outputs
- feasibility criteria
- expected checker behavior
- intended solution idea

State assumptions explicitly. If the user gives an exact construction or exact feasibility list, treat it as authoritative unless it is contradictory. Do not "improve" the math silently.

Stop and ask only when a missing detail changes the problem semantics or output contract.

## Phase 1: Local Components

Create or repair local components in this order:

1. Statement and samples.
2. Standard accepted solution.
3. Validator via `stressor-vali`.
4. Brute/reference oracle via `stressor-brute` when feasible.
5. Checker via `stressor-checker`.
6. Small readable tests via `stressor-gen-small`.
7. Generator suite via `stressor-gen-large`.
8. Wrong/TL/RE solutions when useful for Polygon coverage.

For construction problems with non-unique outputs, the checker must validate the participant output semantically and must not rely on exact diff.

For validators and checkers, allow trailing whitespace after the last meaningful token, but reject extra tokens.

## Phase 2: Local Verification Gates

Run the relevant local checks before Polygon upload:

- compile standard solution, validator, checker, and generators
- validator accepts all samples and generated pilot tests
- validator rejects targeted invalid tests
- standard solution output passes checker on samples and representative tests
- brute/reference output passes checker where runnable
- generator outputs pass validator
- wrong solutions are rejected by at least one local or generated test when they are included as WA/TL/RE coverage
- performance smoke test covers the maximum or near-maximum constraints

If any gate fails, fix the root cause before uploading. Do not weaken tests or checker behavior merely to pass.

## Phase 3: Polygon Upload Plan

Use `polygon-mcp` for remote operations.

Before writes:

- read `get_problem_info`, `get_problem_statements`, `get_problem_tests`, `get_problem_files`, and `get_problem_solutions`
- identify whether the problem is empty or already has user changes
- avoid overwriting remote files silently

Upload and configure in this order:

1. Metadata with `update_problem_info` if needed.
2. Statement with `save_problem_statement`.
3. Chinese statement template if needed: see `references/polygon_release.md`.
4. Source files with `save_problem_file`.
5. Main accepted solution with `save_problem_solution` and tag `MA`.
6. Wrong/TL/RE solutions when applicable.
7. Validator and checker selection.
8. Manual samples only.
9. Generator files and script.
10. Validator tests and checker tests.
11. General tutorial.

All C++ source files for this project should use Polygon source type:

```text
cpp.gcc14-64-msys2-g++23
```

If the MCP schema rejects this value for a specific tool, report the tool limitation and verify the remote `sourceType` by reading `get_problem_files` or the relevant solution list. Do not repeatedly overwrite files with the wrong source type.

## Phase 4: Polygon Test Data Policy

Manual Polygon tests:

- allowed for statement samples
- usually 2 to 4 tests
- may include one extra smoke test only when the user asks

Generated Polygon tests:

- official non-sample tests must come from generator script lines
- default script style is `>$`
- after saving the script, read `view_problem_script`
- read several generated tests with `view_problem_test_input` to confirm they actually expand

If generated tests cannot expand, first check that the main accepted solution is tagged `MA`.

Readiness may warn that script lines are missing when `>$` is normalized by the remote or MCP display layer. Treat this as a warning only after confirming tests are `manual=false`, have script lines, and generated inputs can be read.

## Phase 5: Statement And LaTeX Policy

All local LaTeX content must be XeLaTeX-compatible. If not compiled locally, state that it was not locally compiled.

Polygon statement rendering may use a generated `statements.tex` from `statements.ftl`. For Chinese statements, modify `statements.ftl` as described in `references/polygon_release.md`; do not replace the template with a simplified hand-written one.

For Polygon-safe statement content:

- prefer `\leq` and `\geq` over `\le` and `\ge`
- prefer `$$...$$` over `\[...\]` when Polygon HTML rendering rejects display math
- avoid Markdown backticks in statement fields; use `\texttt{}`
- avoid Markdown lists in statement fields; use LaTeX `itemize` or prose
- avoid unsupported or ornamental macros unless verified by Polygon render/build

For Chinese statement rendering, modify the existing `statements.ftl` template directly, preserving its FreeMarker/import structure and changing only the LaTeX preamble as described in `references/polygon_release.md`. Package READY alone is not sufficient evidence for statement-page rendering.

## Phase 6: Readiness, Commit, Build

Run `check_problem_readiness` before release-sensitive steps.

If blocking issues exist, fix them and rerun readiness.

If warnings exist:

- classify each warning as real, acceptable, or MCP/reporting artifact
- do not ignore missing validator/checker/main solution warnings
- `>$` script-line warnings may be acceptable only after generated input expansion is verified

Use `commit_problem_changes` or `prepare_problem_release` as appropriate.

If `prepare_problem_release` or build polling times out at the client side, do not assume failure. Read remote state with:

- `get_problems`
- `get_problem_packages`
- `check_problem_readiness`

Report the actual revision, modified flag, latest package id, package state, and whether verification was run.

## Phase 7: Final Report

Final response must include:

- local files created or changed
- root causes for any failures fixed during the workflow
- local verification commands actually run
- Polygon object touched
- MCP tools used at a high level
- readiness/build/release status
- remaining warnings or unverified assumptions

Never say "ready" if only local files were created. Never say "verified" without a concrete command or Polygon package result.
