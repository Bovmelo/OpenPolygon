# Polygon Release Notes

Load this reference when `stressor-polygon-release` reaches Polygon upload, statement rendering, readiness, commit, or build steps.

## C++ Source Type

For this project, C++ source files should use:

```text
cpp.gcc14-64-msys2-g++23
```

This applies to validators, checkers, generators, and normal C++ source files when the Polygon tool accepts compiler source types.

Some MCP tools use `source_type` for a different concept, such as solution kind, and may reject compiler strings. In that case:

- follow the tool schema for that operation
- read back remote files or solutions to confirm the actual type/tag
- do not repeatedly reupload files in a way that downgrades the remote compiler type

## Main Accepted Solution

Generated Polygon tests may fail to expand if there is no main accepted solution.

For the primary accepted solution, save it with tag:

```text
MA
```

If an existing accepted solution is only `OK`, resave or retag it as `MA` according to the available MCP tool behavior.

## Manual Versus Generated Tests

Manual tests are acceptable for statement samples. Keep them small and few.

Official non-sample tests should be generated by script lines, normally using:

```text
gen args >$
```

After saving the script:

- call `view_problem_script`
- call `get_problem_tests`
- call `view_problem_test_input` on several generated tests

Only treat the generated data as valid after actual generated input can be read.

Readiness may warn that script lines do not match when the script uses `>$` or when MCP normalizes it as `> $`. This can be accepted only if generated tests are not manual and generated inputs can be read.

## Validator And Checker Tests

Add validator tests:

- legal minimum
- legal maximum
- illegal out-of-range
- illegal structural constraint

Add checker tests:

- one valid accepted output
- one valid impossible `NO` output, if the problem has impossible cases
- one malformed or semantically wrong output
- one output that violates the key semantic condition

When a checker ignores the jury answer, Polygon may still require a non-empty `testAnswer`; use a harmless placeholder such as `unused`.

## Chinese Statement Rendering

Local project rules require XeLaTeX-compatible LaTeX. For Chinese statements, modify `statements.ftl` directly instead of inventing a new simplified template.

Make the generated `statements.tex` use `ctexart`:

```tex
\documentclass [11pt, a4paper, oneside] {ctexart}
```

Use this by modifying the existing problem `files/statements.ftl` directly:

1. Start from the existing Polygon-style `statements.ftl`.
2. Change only the document class from `article` to `ctexart`.
3. Preserve the original package list, FreeMarker conditions, `\contest`, `\graphicspath`, and `\import` / `\input` structure.
4. Optionally add these formatting lines when appropriate:

```tex
\usepackage {float}
\setlength{\parindent}{2em}
```

The useful problem-level template shape is:

```tex
\documentclass [11pt, a4paper, oneside] {ctexart}

\usepackage [T2A] {fontenc}
\usepackage [utf8] {inputenc}
\usepackage [english, russian] {babel}
\usepackage {amsmath}
\usepackage {amssymb}
\usepackage <#if contest.language?? && contest.language="russian">[russian]<#elseif contest.language?? && contest.language="ukrainian">[ukrainian]</#if>{olymp}
\usepackage {comment}
\usepackage {epigraph}
\usepackage {expdlist}
\usepackage {graphicx}
\usepackage {multirow}
\usepackage {siunitx}
\usepackage {ulem}
\usepackage {import}
\usepackage {ifpdf}
\usepackage {xparse}
\usepackage {float}
\ifpdf
  \DeclareGraphicsRule{*}{mps}{*}{}
\fi

\setlength{\parindent}{2em}

\begin {document}

\contest
{${contest.name!}}%
{${contest.location!}}%
{${contest.date!}}%

\binoppenalty=10000
\relpenalty=10000

\renewcommand{\t}{\texttt}
\renewcommand{\thefootnote}{\fnsymbol{footnote}}

<#if shortProblemTitle?? && shortProblemTitle>
  \def\ShortProblemTitle{}
</#if>

<#list statements as statement>
<#if statement.path??>
\graphicspath{{${statement.path}}}
<#if statement.index??>
  \def\ProblemIndex{${statement.index}}
</#if>
\import{${statement.path}}{./${statement.file}}
<#else>
\input ${statement.file}
</#if>
</#list>

\end {document}
```

Do not replace the template with a minimal hand-written loop over statements. The full template structure matters because Polygon/contest packaging expects its FreeMarker variables, imported problem files, examples, resources, and style hooks to stay intact.

After editing, upload the modified `statements.ftl` as a problem file/resource through Polygon MCP, read it back, then commit and build.

Verification signals:

- A successful XeLaTeX-style log includes `ctexart.cls`, `xeCJK.sty`, and Fandol font loading.
- If the renderer still logs `Document Class: article` from `statements.tex`, the modified template is not being used.
- Package READY alone is not enough to prove the statement page renderer works. Treat user-provided statement render logs as authoritative evidence.

## Polygon-Safe LaTeX Subset

Use conservative statement syntax:

- `\leq`, `\geq`
- `$$...$$` if `\[...\]` is rejected by Polygon HTML rendering
- `\texttt{...}` for literal tokens
- `itemize` or prose instead of Markdown lists
- avoid unsupported display macros unless verified

When editing statement content, update all language slots that intentionally carry the same Chinese body.

## Build And Release Timeout

MCP tool calls can time out while the remote operation continues.

After a timeout:

- call `get_problems`
- call `get_problem_packages`
- call `check_problem_readiness`

Report actual remote state instead of assuming failure:

- revision
- modified flag
- latest package id
- package state
- whether the package was built with verification

## Readiness Warning Policy

Blocking issues must be fixed before release.

Warnings require classification:

- real warning: fix or explain risk
- acceptable warning: explain why
- MCP/reporting artifact: prove with remote reads

Examples:

- missing wrong solutions can be acceptable for a simple problem, but should be stated
- `>$` script mismatch can be an artifact only after generated inputs are readable
- missing statement sample verification should be fixed when sample outputs are known
